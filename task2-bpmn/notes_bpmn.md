# Конспект по нотации BPMN 2.0

## 1. Назначение BPMN

BPMN (Business Process Model and Notation) — международный стандарт
графического описания бизнес-процессов, разработанный организацией OMG
в 2004 году, актуальная версия 2.0 принята в 2011 году. В отличие от
произвольных блок-схем, каждый элемент BPMN имеет строгое семантическое
значение, что обеспечивает однозначное прочтение диаграммы любым
специалистом. Применяется для описания оркестровки процессов, взаимодействия
участников, диалогов и хореографии обмена сообщениями.

## 2. Таблица элементов нотации

| Элемент | Обозначение | Значение | Пример |
|---|---|---|---|
| Start Event | Тонкий круг | Начало процесса | Получена заявка клиента |
| End Event | Жирный круг | Завершение процесса | Кредит выдан |
| Intermediate Event | Двойной круг | Событие внутри процесса | Ожидание ответа 3 дня |
| Операция (Task) | Прямоугольник со скруглёнными углами | Атомарное действие | Проверить документы |
| Интерактивная операция | Прямоугольник с иконкой человека | Задача, выполняемая человеком через систему | Заполнить анкету в CRM |
| Ручная операция | Прямоугольник с иконкой руки | Задача без ИТ-поддержки | Подписать договор |
| Сервис | Прямоугольник с шестерёнкой | Автоматическая задача | Отправить SMS-уведомление |
| Подпроцесс | Прямоугольник с «+» внизу | Составная операция | Проверка кредитоспособности |
| XOR (Исключающее ИЛИ) | Ромб с X | Только одна ветка из нескольких | Одобрено / Отказано |
| AND (И) | Ромб с «+» | Все ветки выполняются параллельно | Уведомить все службы одновременно |
| OR (ИЛИ) | Ромб с кругом | Одна или несколько веток | Уведомить по email и/или телефону |
| Pool | Большой прямоугольник | Участник процесса | Банк, Клиент |
| Lane | Дорожка внутри пула | Роль или подразделение | Операционист, Менеджер |
| Sequence Flow | Сплошная стрелка | Поток управления внутри пула | → |
| Message Flow | Пунктирная стрелка | Поток сообщений между пулами | Клиент → Банк |
| Аннотация | Скобка с текстом | Пояснение к элементу | Срок рассмотрения — 3 дня |

## 3. Типы шлюзов

| Шлюз | Условие применения | Пример | Типичная ошибка |
|---|---|---|---|
| XOR (Исключающее ИЛИ) | Только одна ветка — взаимоисключающие условия | Заявка одобрена / отклонена / недостаточно данных | Не подписывать условия на исходящих стрелках |
| AND (И) | Все ветки выполняются одновременно | Уведомить службу безопасности, бухгалтерию и юриста параллельно | Забыть поставить AND-шлюз слияния в конце |
| OR (ИЛИ) | Одна или несколько веток по условию | Уведомить клиента по email и/или SMS | Путать с XOR, когда условия не пересекаются |
| Событийный | Первое наступившее событие определяет ветку | Ответ получен / истёк таймер 3 дня | Использовать вместо XOR для логических условий |

## 4. Типы подпроцессов

**Collapsed Sub-Process (свёрнутый)** — отображается как прямоугольник
с «+» внизу. Детали скрыты и раскрываются на отдельном листе диаграммы.
Используется когда подпроцесс сложный и его детализация загромождала бы
основную диаграмму.

**Expanded Sub-Process (вложенный/раскрытый)** — все задачи внутри видны
прямо на диаграмме. Используется когда подпроцесс простой (2–3 задачи).

**Событийный подпроцесс** — запускается только при наступлении определённого
события (получение сообщения, ошибка). Живёт только пока жив родительский
процесс. Обозначается пунктирной границей.

**Выбор типа:** если подпроцесс содержит более 3 задач или имеет
собственные ветвления — используй Collapsed. Если простой — можно Expanded.

## 5. Типы диаграмм BPMN

- **Оркестровка** — описывает внутренние процессы участника (закрытый
  процесс, аналитическая схема, исполняемая схема)
- **Взаимодействие** — показывает обмен сообщениями между несколькими
  участниками через Message Flow
- **Диалоги** — описывает переговоры между участниками
- **Хореография** — описывает регламент обмена сообщениями

## 6. Граничные события (Boundary Events)

Прикрепляются к границе задачи или подпроцесса и срабатывают при
наступлении события во время выполнения задачи:

- **Таймер** — срабатывает по истечении времени. Прерывающий останавливает
  задачу, не прерывающий запускает параллельную ветку
- **Ошибка** — срабатывает при возникновении ошибки, всегда прерывающий
- **Эскалация** — что-то нехорошее, но не критичное; может быть
  не прерывающим

## 7. Авторский чек-лист проверки диаграммы

1. У каждого пула есть ровно одно Start Event
2. Каждая завершающая ветка заканчивается своим End Event с подписью
3. Sequence Flow не пересекает границу пула — только Message Flow между пулами
4. На каждой исходящей стрелке шлюза XOR подписано условие
5. После AND-шлюза разветвления обязательно стоит AND-шлюз слияния
6. Каждый Collapsed Sub-Process раскрыт на отдельном листе
7. Все задачи именованы глаголом: «Проверить», «Отправить», «Согласовать»
8. Каждая стрелка начинается и заканчивается на конкретном элементе
9. Message Flow используется только между разными пулами
10. Граничные события таймера используются для отражения ожидания и дедлайнов
